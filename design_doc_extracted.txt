Orchestra — End-to-End Backend Architecture
Stack Decisions
Runtime: Node.js (TypeScript) with Fastify or Express
AI Layer: Anthropic Claude API (Sonnet 4) via direct calls
Vector DB: Pgvector (Postgres extension) — keeps infra unified
Primary DB: PostgreSQL (Supabase or self-hosted)
Cache: Redis (session, rate limits, doc chunk hot cache)
File Storage: S3-compatible (Supabase Storage or AWS S3)
Queue: BullMQ (Redis-backed) for async doc ingestion
Auth: JWT + refresh tokens, role-based (PM/Dev/Client)
Realtime: Supabase Realtime or Socket.io for live doc viewer
Database Design
organisations
id, name, slug, created_at
users
id, org_id (FK), name, email, password_hash, role (enum: pm | dev | client), created_at
projects
id, org_id (FK), name, description, status (enum: active | paused | complete), created_at
project_members
id, project_id (FK), user_id (FK), role_in_project, joined_at
documents
id, project_id (FK), uploaded_by (FK → users), title, file_key (S3 path), file_type (pdf/docx/txt), status (enum: pending | processing | ready | failed), created_at
document_chunks ← Brain/RAG backbone
id, document_id (FK), chunk_index, content (text), embedding (vector 1536), page_number, token_count, created_at
socrates_sessions
id, user_id (FK), project_id (FK, nullable), page_context (e.g. "dashboard" | "brain" | "flow"), created_at
socrates_messages
id, session_id (FK), role (enum: user | assistant), content, cited_chunk_ids (uuid[]), suggested_prompts (jsonb), created_at
tasks
id, project_id (FK), assignee_id (FK → users), title, description, status (enum: planned | in_progress | review | done), priority, source(enum: manual | socrates | jira_sync), external_id (Jira/GitHub ref), created_at, updated_at
task_events ← audit trail / dashboard feed
id, task_id (FK), triggered_by (FK → users), event_type, old_value (jsonb), new_value (jsonb), created_at
features
id, project_id (FK), name, status, doc_reference_ids (uuid[]), created_at
dashboard_snapshots ← precomputed for perf
id, project_id (FK, nullable), org_id (FK), scope (enum: project | general), payload (jsonb), computed_at

POST /

Realtime /ws
Socket rooms per project_id — broadcasts task updates, doc processing status, Socrates streaming tokens
Background Workers (BullMQ)
doc-ingestion queue
Pull file from S3
Parse (pdf-parse / mammoth / raw txt)
Chunk by token window (512 tokens, 50 overlap)
Embed each chunk via text-embedding-3-small (OpenAI) or Voyage (cheaper)
Upsert into document_chunks with pgvector
Update documents.status = 'ready'
Emit realtime event to frontend
snapshot-refresh cron (every 15 min)
Recompute dashboard_snapshots for all active projects
Invalidate Redis cache keys
suggestion-precompute worker
Triggered on page navigation event
Pulls top-k chunks relevant to current page context
Precomputes 3–5 Socrates suggestions, stores in session
Socrates RAG Pipeline (per message)
Receive user_message + session_id
Fetch page_context + project_id from session
Embed user message → query_vector
pgvector similarity search: SELECT * FROM document_chunks WHERE project_id = ? ORDER BY embedding <=> query_vector LIMIT 8
Build context string from top-k chunks (include document_id, page_number for citations)
Construct Claude prompt: system (role + constraints) + context + conversation history (last 10 turns) + user message
Stream response back via SSE/WebSocket
Parse response for cited_chunk_ids → store in socrates_messages
Frontend uses cited_chunk_ids to highlight + navigate to exact doc page (Mannan's feature)
Infra / Deployment
API server: Railway or Render (Node.js container)
DB: Supabase (Postgres + pgvector + Realtime + Storage in one)
Queue: Upstash Redis (BullMQ)
Worker: Separate Railway service for ingestion worker
Env secrets: Doppler or Railway env vars
CDN: Cloudflare in front of API + S3 signed URLs with 1hr TTL
Monitoring: Sentry (errors) + Axiom or Logtail (structured logs)
Key Design Decisions
Pgvector over Pinecone — keeps vector search and relational queries in one DB, simplifies joins (e.g. "which tasks reference this chunk")
Snapshot pattern for dashboard — avoids expensive real-time aggregation on every load; refreshes every 15 min with a delta layer for live updates
Page context on Socrates — session stores current page_context string so suggestions are scoped to what the user is looking at, not generic
Chunk-level citations — every Socrates response stores which chunk IDs it used, enabling the click-to-source doc navigation Mannan proposed
BullMQ over Lambda — more predictable for long-running ingestion jobs, retries built-in, visible queue dashboard
